Staging Edit — User Journeys & Page State Diagram

LifeTime Deployment Plan Editor · Analysis for UI porting

User Journeys & Jobs to Be Done
Journey 1
Build a new deployment plan
"When I need to ship a new release, I want to assemble a list of apps and target versions in one place, so that I have a clear and repeatable plan before anything runs."
Steps
  • 1Enter the Edit page from a new or saved draft plan
  • 2Add applications one by one or bulk-add all
  • 3Review the default deployment option selected per app
  • 4Optionally change version or skip individual apps
  • 5Save the plan for later or proceed to validation
→ Save Plan → Validate Now
Journey 2
Validate a plan before deploying
"When my plan is assembled, I want to run a validation so I can catch dependency conflicts and version mismatches before they cause runtime errors in production."
Steps
  • 1Plan is in NeedValidation state
  • 2Click "Validate Now"
  • 3Wait for validation to complete ("Validating…")
  • 4Review per-app results in each row
  • 5If all OK → Continue to Confirm step
→ Continue → Fix errors first
Journey 3
Resolve warnings before deploying
"When validation returns warnings on some apps, I want to understand what's at risk and decide whether to adjust options or accept the risk, so that I deploy intentionally."
Steps
  • 1Validation completes with warnings on one or more rows
  • 2Expand the affected row's options panel
  • 3Click "Show Details" to open the file diff / comparison popup
  • 4Optionally switch to a different version option
  • 5Re-validate, then continue
→ Re-validate → Continue with warnings
Journey 4
Fix or override validation errors
"When validation returns hard errors, I want to understand what's broken and either fix the plan or consciously force-deploy, so that I'm not blindsided at execution time."
Steps
  • 1Validation completes with errors — action bar shows "Errors" state
  • 2Click "Fix Errors in Selection" — page scrolls to first errored row
  • 3Expand row, read ModuleInconsistencyMessage / InconsistencyType
  • 4Either: change option (re-validate) or accept error
  • 5If errors remain: "Continue with errors" becomes available
→ Re-validate after fix → Continue with errors
Journey 5
Save and resume later
"When I'm not ready to deploy now, I want to save my plan in progress so I can come back later without losing my app selection and option choices."
Steps
  • 1Build or adjust the plan (apps, options)
  • 2Click "Save Plan"
  • 3Page stays open — plan is persisted
  • 4User can return later via the Stagings List
→ Stay on page → Leave & return later
Journey 6
Discard a plan
"When a plan is no longer needed or was created by mistake, I want to discard it cleanly so I can start fresh without clutter."
Steps
  • 1Click "Discard Plan"
  • 2Confirm dialog: "Discard the current plan?"
  • 3Plan is deleted
  • 4Redirected to Stagings List
→ Stagings List
Journey 7
Understand why the page is blocked
"When I open a plan and can't do anything, I want to know exactly why I'm blocked so that I know what to fix or when to come back."
Steps
  • 1Page loads — error banner shown at top
  • 2Either: source/target environment in maintenance mode
  • 3Or: ongoing publish in target env (with "here" link to more info)
  • 4All actions (Continue, Save, Add app) are disabled
  • 5User must wait or fix the environment condition
→ Wait / fix env
Journey 8
Add more apps to an existing plan
"When a deployment plan already exists with some apps, and I realise another app also needs to go out, I want to add it to the existing plan so I don't have to start over or run two separate deployments."
Steps
  • 1Open an existing draft plan — already has apps and may have prior validation results
  • 2Click "Add Applications" to open the app picker
  • 3Select the additional app(s) and confirm
  • 4New rows appear in the list alongside existing ones — plan enters NeedValidation state
  • 5Previous validation results on existing rows are now stale — entire plan must be re-validated
  • 6Click "Validate Now" to re-validate all apps together
  • 7Review results across all rows (old and new)
  • 8Save or continue to deploy
Key distinction from J1: existing app rows and their options are preserved. Adding a new app invalidates the whole plan's validation state, not just the new row.
→ Validate Now → Save Plan → Continue
