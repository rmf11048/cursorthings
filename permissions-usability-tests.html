Permissions UI — Usability tests

Simulated test scenarios for administrators

Use these tasks during moderated or unmoderated sessions. Give the participant the prototype and read each scenario and task. Observe whether they complete the task, where they hesitate, and whether they need help. You can time each task and note errors or recovery.

Task 1 — Restrict Production access

T1 · ~1 min

Restrict Production to “No access”

“Your team has decided that this role should not have any access to Production. Update the permissions so that Production has no access.”

  1. Participant opens or is already on the Permissions prototype.
  2. Participant finds the Production column.
  3. Participant checks “No access” for Production (or clears other permissions until only No access applies).

Success: Production column shows only “No access” checked; no permission levels or Create applications / Add system dependencies are selected.

Task 2 — Grant “Create applications” in QA

T2 · ~2 min

Enable Create applications for QA

“QA needs to be able to create applications in the QA environment. Set the QA column so that they can create applications there.”

  1. Participant finds the QA column.
  2. Participant sets the permission level to “Change and deploy applications” or “Full control” (required for Create applications).
  3. Participant checks “Create applications” for QA.

Success: QA has at least “Change and deploy applications” (or Full control), and “Create applications” is checked for QA.

Task 3 — Set “Open and debug” for QA

T3 · ~1 min

Set QA to “Open and debug applications” only

“For QA we only want this role to open and debug applications, not change or deploy. Set QA to that level.”

  1. Participant finds the QA column.
  2. Participant selects “Open and debug applications” (and understands that this and the rows below are checked, not Full control or Change and deploy).

Success: In QA, “Open and debug applications” is the highest checked row in the permission list; “Full control” and “Change and deploy applications” are unchecked.

Task 4 — Add system dependencies in Development

T4 · ~1 min

Allow adding system dependencies in Development

“Developers should be able to add system dependencies in the Development environment. Enable that for Development.”

  1. Participant ensures Development has at least “Access” (any permission level is enough).
  2. Participant checks “Add system dependencies” for Development.

Success: “Add system dependencies” is checked for Development (and Development is not “No access”).

Task 5 — Understand why Create applications is disabled

T5 · ~1 min

Find out why Create applications is disabled

“In the current setup, ‘Create applications’ is disabled for QA. Without clicking it, find out why it’s disabled and what would be needed to enable it.”

  1. Participant hovers or focuses the disabled “Create applications” cell for QA (or reads the UI).
  2. Participant sees the tooltip or infers that a higher permission level is required (e.g. “Change and deploy applications or above”).

Success: Participant correctly states that a higher permission level (e.g. Change and deploy or Full control) is required, using the tooltip or the hierarchy.

Task 6 — Full setup from scratch

T6 · ~3 min

Configure all three environments

“Configure permissions so that: Development has full control and can create applications; QA has open and debug only and cannot create applications; Production has no access.”

  1. Participant sets Development to Full control and checks Create applications (and optionally Add system dependencies).
  2. Participant sets QA to “Open and debug applications” (Create applications should be disabled/unchecked).
  3. Participant sets Production to No access.

Success: Development = Full control + Create applications; QA = Open and debug only, Create applications not selected; Production = No access only.

Notes for facilitators

  • Reset the prototype to the initial state between participants (or document the starting state).
  • For T2 and T5, watch whether participants discover the level requirement and the tooltip.
  • Record time-on-task, success rate, and any verbal feedback about the hierarchy or the “Create applications” / “Add system dependencies” rules.
